Redis 延迟双删真的可靠吗？缓存一致性 8 种方案对比 ​

这篇不是“标准答案”，而是我在项目里踩过坑之后的一份实战笔记。

先说结论（给赶时间的你） ​

延迟双删有用，但它不是银弹。

  • 写入量不大、对“短暂不一致”能接受：延迟双删够用
  • 对一致性要求高（库存、余额、名额）：别只靠延迟双删，至少加重试兜底
  • 链路很复杂（多服务、多语言、多写入口）：考虑 Binlog 驱动或消息驱动的一致性方案

一句话：延迟双删是“便宜且有效”的工程折中，不是“绝对正确”的理论方案。

为什么会有“延迟双删”这个玩法？ ​

我们一般会用 Cache-Aside：

  1. 读：先读 Redis，没命中再查 MySQL，再回填缓存
  2. 写：先写 MySQL，再删 Redis

这套没问题，但并发下会出现一个经典时序：

于是延迟双删登场了：

  1. 更新数据库
  2. 删除缓存
  3. sleep 一小段时间
  4. 再删一次缓存

第二次删除的目的很朴素：把“并发窗口里被回填的脏缓存”再清掉一次。

但它为什么经常被喷“不可靠”？ ​

因为它确实有边界，常见翻车点有 4 个：

  1. 延迟时间不好设 延迟太短，脏数据还没来得及回填；太长，又放大了不一致窗口。

  2. 第二次删除丢了 线程池满了、任务被丢弃、服务重启、GC 暂停，都可能导致第二删没执行。

  3. 多写入口不统一 A 服务写库删缓存，B 服务直接改库不删缓存，规则一旦不统一，方案形同虚设。

  4. 主从复制延迟 读走从库时，可能先读到旧数据再回填，双删只能缓解，不能消灭主从延迟。

所以说它不是“错”，而是需要配套措施。

8 种一致性方案，放一张表里讲清楚 ​

方案一致性强度实现复杂度性能成本我的建议
先更新缓存再更新数据库低低低不建议，用并发很容易乱
先更新数据库再更新缓存中中中需要严格串行与重试，不常用
先更新数据库再删缓存中（最终一致）低低默认首选
延迟双删中偏上（最终一致）低低适合中小型高并发场景
删缓存失败重试（MQ）中偏上（最终一致）中中强烈建议和双删组合
分布式锁串行写高中中偏高关键写路径可用，别全量上锁
Binlog 订阅（Canal/Debezium）中偏上（最终一致）高中系统大了值得上
读写穿透（不走缓存写）+ 短 TTL中低中读多写少场景挺实用

一个“能落地”的推荐组合 ​

我自己更推荐这套，而不是只讲延迟双删：

  1. 主流程：先写库，再删缓存
  2. 兜底：发一条“删缓存消息”到 MQ（失败重试、死信告警）
  3. 热点 key：加互斥重建（防击穿）
  4. 过期策略：TTL 加随机值（防雪崩）
  5. 监控：缓存删除失败率、回源 QPS、脏读告警

这套的优点是：即使某一步偶发失败，也有后手，不会把一致性赌在线程 sleep 上。

代码示例：延迟双删 + MQ 重试 ​

java
@Service
public class ProductService {

    @Resource
    private ProductMapper productMapper;
    @Resource
    private StringRedisTemplate redisTemplate;
    @Resource
    private CacheDeleteProducer cacheDeleteProducer;
    @Resource
    private ScheduledExecutorService scheduledExecutorService;

    private static final long SECOND_DELETE_DELAY_MS = 500L;

    @Transactional(rollbackFor = Exception.class)
    public void updateProduct(Product product) {
        String cacheKey = "product:" + product.getId();

        // 1) 先更新数据库（以 DB 为准）
        productMapper.updateById(product);

        // 2) 第一删：立即删缓存
        redisTemplate.delete(cacheKey);

        // 3) 第二删：延迟再删一次，清理并发窗口中的脏回填
        scheduledExecutorService.schedule(
            () -> redisTemplate.delete(cacheKey),
            SECOND_DELETE_DELAY_MS,
            TimeUnit.MILLISECONDS
        );

        // 4) 兜底：异步消息重试删除（应对线程池丢任务、服务重启等情况）
        cacheDeleteProducer.send(cacheKey);
    }
}

消费者侧建议最少做三件事：

  1. 幂等删除（删不存在 key 也算成功）
  2. 指数退避重试（比如 1s、5s、30s）
  3. 超过阈值进死信队列并告警

延迟时间到底怎么取？ ​

很多文章一上来就给“500ms”，但这个值不是玄学常量。

可以按这个思路估：

延迟时间 >= 一次读请求完成“查库+回填缓存”的 P99 耗时

举例：

  • 查库 P99：120ms
  • 网络抖动 + 序列化 + Redis 写入：80ms
  • 预留裕量：100ms

那你可以先从 300ms 起步，在压测里看脏读比例再调。

不要拍脑袋，也别一刀切上 2 秒。延迟越长，一致性窗口未必更优，反而可能拉高业务感知延迟。

什么时候不要用延迟双删？ ​

下面三类，我建议直接升级方案：

  1. 强一致业务：余额、库存、优惠券名额
  2. 跨服务多写入口：很难保证每个入口都遵守同一删缓存规范
  3. 审计要求高：需要可追溯、可回放的一致性链路

更稳的选择是：数据库事务完成后，统一发事件，再由独立消费者处理缓存变更。

我在项目里用过的排障清单 ​

线上出现“缓存是旧的”时，我通常按这张单子排：

  1. 是否存在绕过服务层直写数据库的入口
  2. 缓存 key 规则是否统一（有没有大小写、前缀不一致）
  3. 第二次删除任务是否执行成功（线程池拒绝、任务异常）
  4. MQ 删除消息是否堆积/死信
  5. 是否读了从库且主从延迟升高
  6. 热点 key 是否触发击穿并大量回填

很多时候，不是方案错，而是工程细节没兜住。

最后总结 ​

延迟双删不是“神技”，但它是一个非常实用的工程工具。

我的建议是：

  1. 把它当作“基础动作”
  2. 再配一个“异步重试删除”的兜底
  3. 用监控把一致性问题可视化

这样你会得到一个既不重、又不脆的缓存一致性方案。

真正靠谱的系统，从来不是靠某一个技巧，而是靠“一组互相兜底的机制”。
