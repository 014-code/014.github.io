为什么HashMap的默认负载因子是0.75 ​

  1. 负载因子表示 当前容量 * 负载因子的值，一但到达它就触发扩容，太低的话扩容太频繁，浪费性能，太高的话每个桶里面的元素太过密集，查找效率就变低了，冲突也会变高，0.75就是折中值
  2. 在cpu性能较差设备上运行时可以适当调高，在高频交易系统的热点缓存下可以适当调低降低冲突

如何处理mysql的主从延迟 ​

  1. mysql的主从延迟必然存在，只能尽量去降低其主从延迟带来的问题
  2. 写后立即需要读的场景走主库，比如注册登录，虽然失去了一定程度上的读写分离，但是在一致性上会比较强，且此类操作频率不高
  3. 写操作后记录时间戳，短时间内的请求走主库，过了时间窗口再走从库
  4. 二次请求兜底，从库查完没结果再查主库，恶意查询不存在数据主库会被干爆
  5. 缓存前置，主库写完缓存入redis，再读redis，但是这样又需要解决缓存与数据库的一致性
  6. 主从库性能配置问题

HashMap的扩容机制是什么样的 ​

  1. 当前容量 * 负载因子的值，一但到达它就触发扩，且是翻倍扩，2的幂次
  2. 扩容后每个元素重新计算位置并分配移动到新的数组中，扩容就是按(n - 1) * hash，扩容次数减一 * 原本的hash值然后移动

Redis为何设计成单线程且6.0后为何引入多线程 ​

  1. redis的性能瓶颈不在cpu，而在网络io与内存
  2. 数据从内核拷贝到用户空间这一步是同步的，且耗时，会造成性能瓶颈，所以6.0后将这些网络io操作交由多线程处理
  3. 其实4.0后就引入了后台线程来执行大key删除、RDB快照和AOF日志记录这些耗时操作
  4. 主线程还是负责命令执行，命令执行不用多线程是因为redis内置的如ZSet本身就是线程不安全的，多线程会带来额外负担

为什么TCP挥手需要有TIME_WAIT状态 ​

  1. 为了保证连接可靠关闭：如果在客户端收到FIN后发送的ACK丢了，那么此时客户端已经关闭跑路closed，服务端会收到丢失从而报错，TIME_WAIT的话就是要等服务端接收到ACK，服务端重发ACK才算有效，就是告诉他我收到了，你放心关闭把
  2. 为了防止旧数据对新连接产生影响：旧连接关闭后某些数据包在网络传输中有一定延迟，此时又在原来的ip端口上新建了一个连接导致旧数据传输到新连接中了
  3. 时间2msl保证我发的ACK到达对方，和对方重发的FIN到达我

为什么HashMap扩容采用2的n次方倍 ​

  1. 因为采用(n - 1) & hash，比如n=16，那么位运算时，15转成的二进制就是00001111，低四位全是1，& hash运算一下就可以直接拿来当下标使用
  2. 其实好处就是位运算来代替取模运算，取模运算cpu做除法比较慢，扩容时也如上面那样方便迁移
  3. 为什么n - 1的地位必须全是1：如果n = 10，那么二进制就是1001，中间两个永远出不来1，也就是下标2、3、6、7这些永远映射不到，数组效率直接减半
  4. 如果你在new HashMap指定容量为10，那他会自动向上取最近的2次方为16

TCP超时重传是为了解决什么问题 ​

  1. 为了解决网络传输过程中的丢包问题，在发出数据后新建一个定时器，如果在规定时间内没收到ACK请求则默认丢包了，就会重发

mybatis插件的运行原理 ​

  1. 就是动态代理，在sql执行的关键点进行拦截并增强功能
  2. 包含一下四种：一个负责执行增删改查操作、一个负责解析sql语句的参数、一个负责解析sql语句，一个负责处理结果集

数组和链表的区别 ​

  1. 数组是在内存上连续的，且大小相对固定，对于随机访问数组中某个元素时较块，但是插入/修改时要维护其它位置效率较低，因此适合读多写少的操作
  2. 链表在内存上是分散的，且大小不一，靠的是指针将这些大小不一的串起来，对于修改其中某个位置元素较快，但是随机访问某个元素较慢，因为要重头/尾遍历查找，因此适合修改操作较多场景

redis常见的数据类型 ​

  1. String：字符串常用于页面数据，计数等场景
  2. Hash：键值对集合
  3. List：有序字符串集合，底层用双向链表，常用实现消息队列
  4. Set：不重复且无序的集合，查找去重效率较高
  5. ZSet：Set的特殊类型，加了score进行排序，多用于排行榜
  6. BitMap：位存储，内存占用极低
  7. GEO：底层是ZSet，多用于地理位置存储，经纬度会被编码为score
  8. HyperLogLog：占用固定内存大小，但是有误差代价，适用于统计网站
  9. Stream:专门用于消息队列的，特性是能生成全局唯一消息id，支持消费组模式，消费者挂了重启后还能进行消费

TCP滑动窗口的作用 ​

  1. 就是做流量控制，发送方不能一股脑把数据包都发给接收方缓冲区都炸了
  2. 还有就是能够在滑动窗口的时间内连续发多个包，不用傻等ACK回来再发下一个包
