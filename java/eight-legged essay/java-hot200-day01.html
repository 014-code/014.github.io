HashMap的原理 ​

  1. 本质上是个数组，结合链表和红黑树来解决冲突
  2. 怎么存：会先计算key的hashCode根据它来通过(table.length - 1) & hash来计算在数组下标中的位置
  3. hash冲突：就是不同的key算出来下标一样了，jdk8之后对于冲突的用链表将其连起来，大小超过8自动转为红黑树
  4. 扩容机制：默认有个负载因子：0.75，就是数组用了75%后会自动大小翻倍，然后把所有数据重新计算位置加到新数组里面rehash，比较消耗性能，所以最好一开始就确定预估初始大小
  5. 添加元素先查询当前hashCode位置元素key是否和要加入的相同，相同就替换，不同要判断是哪种结构红黑树的话则添加元素，链表的话要看个数是否超过8且table大小超过64就转为红黑树，否则就扩容table
  6. 第一次添加table容量为16，后添加一次翻倍

ConcurrentHashMap1.7和1.8的区别 ​

  1. 1.7的话是基于分段锁的，被分成16块区域，每块区域一把锁,不同线程只访问不同区域，16块区域占满才开始竞争锁，每个区域都是一个HashMap
  2. 1.8的话数据结构发生变化，这些区域被移除，锁颗粒化到每个槽位，换成HashMap一样的数组 + 链表 + 红黑树，插入元素时先CAS强行占位，冲突了再synchronized，且只锁链表和红黑树的头节点，其它bucket槽位照样可以访问
  3. 扩容的话，1.7版本要先把每个区域都上锁在更新size，1.8的话每个槽位都有自己的计数器单元，然后再同一加起来，性能比1.7的高

为什么jdk1.8后对HashMap做了红黑树的改动 ​

  1. 如果出现大量hash冲突的情况下某些槽位下的链表会被不断加长，导致查询变慢，还有可能攻击者会制造大量hash冲突的key，导致链表超长，查询直接卡死
  2. 红黑树的话在数据量大的情况下查询相对比链表要快，变成红黑树两个条件，1.链表长度大于82.数组自身长度大于64
  3. 红黑树节点小于6时会自动退化到链表
  4. 红黑树占用内存比较大，需要额外存储父节点、左右节点、颜色标记等，在数据量小的情况下就没必要反而大的情况下查询就有优势

1.8对HashMap还做了哪些改动 ​

  1. hash码改成高16位和低16位进行异或
  2. 头插法改尾插法，避免多线程造成环问题
  3. rehash扩容改成2的次方
  4. 先插入再看是否大于阈值，然后扩容

mysql的最左前缀匹配原则是什么 ​

  1. 就是在使用联合索引时会按照索引顺序从左到右依次匹配条件
  2. 最优的情况就是按照联合索引顺序来写查询条件且带有其中的一个条件，不然的话联合索引就不会被使用到

脏读、不可重复读、幻读是什么 ​

  1. 脏读是，你读到的是别人还没提交的数据，万一它回滚了你读到的就是不存在的数据
  2. 不可重复读就是同一事务中两次去独这一行数据，因为中间有其它事务将其修改了，导致两次的结果不同
  3. 幻读主要强调读到的行数不同，执行相同的查询范围情况下有其它事务将其修改了
  4. innoDB如何解决这三个问题，1.mvcc只让事务去读快照里面的，而快照里面的本来就只存已提交的数据，2.可重复读级别下，后续查询复用同一个读取结果就不会出现两次读到的内容不一样3.读mvcc的快照，看不到插入的行数据，用间隙锁锁住读取的范围其它事务没法进来

Mysql几大存储引擎的区别 ​

  1. innoDB支持事务，行级锁、MVCC，适合对读写要求高的场景,数据按聚簇索引组织
  2. MyISAM不支持事务，只支持表锁，读性能较强，适合读多写少的场景，对一致性要求不高的，数据按照非聚簇索引组织
  3. MEMORY存储在内存中的，服务器重启后就会丢失，适合做临时数据存储和快速缓存

mysql的索引覆盖是什么 ​

  1. 本质上是一种查询的优化方式，即二级索引中已经包含了查询所需的所有字段，就不用回表到一级索引去查询获取数据，直接从二级索引中获取
  2. 好处是能够减少磁盘io次数，省掉回表的次数，避免额外的性能开销
  3. 减少内存的占用，索引数据存储在内存当中，且读取速度更快

mysql的索引 ​

  1. 从数据结构来看：B+树索引，也是mysql的主要索引，哈希索引、全文索引：类似倒排索引、空间索引：主要用来存储地理位置信息之类的
  2. 从innoDB的存储方式来看分为聚簇和非聚簇，聚簇有主键索引之类的，非聚簇就是二级索引
  3. 从索引类型看有：主键索引、普通索引、唯一索引、联合索引、全文索引、空间索引
  4. 为啥B+树主力：数据库查询的话多为等值/范围，等值hash比较合适，但是范围/等值都合适的只有B+树了，且这种结构在树高比较矮的情况下也能够存储大量数据，且磁盘的io次数也比较少
  5. 聚簇索引存完整行数据，非聚簇只存和索引关联字段行数据和主键值，如果要查的多于非聚簇的话，就要带着id回查聚簇里面的完整数据了

mysql的索引下推是什么 ​

  1. 是一种减少回表次数提升查询速度的方式，把原先部分查询条件从service层下发到引擎层，在引擎层就过滤掉不符数据，然后再返回符合的id给到service再从一级索引拿到完整数据
  2. 因为原先是全部都在service里面完成，先回表拿到完整数据，在过滤，导致即使不符合条件的也得回表
  3. 虽然有的条件不符合最左前缀匹配原则，但是innoDB能过滤掉那个不符的

聚簇与非聚簇的区别 ​

  1. 聚簇的叶子节点存的完整行数据，非聚簇的叶子节点存储主键和索引列的数据
  2. 非叶子节点存储主键和页号，方便直接定位到叶子节点，叶子节点间用链表连接，方便范围顺序查找
  3. 访问数据时一个要回表一个直接查，维护成本主键更新时聚簇的数据物理位置会移动，而非聚簇的话主键不变基本不会改变
  4. 所有聚簇会影响数据物理存储顺序，而非聚簇就不会影响

mysql的回表是什么 ​

  1. 简单来说就是非聚簇/二级索引列不包含全部的查询条件，这要回到聚簇索引中查
  2. 因为非聚簇拿到的id不是连续的导致会带来大量随机io

索引一定有效吗，如何排查索引效果 ​

  1. 不一定有效，在执行的sql前加上explain进行，返回的数据有这些，如果是ref/等值匹配，all基本是全表扫描没走索引，range/范围，index/全索引扫描，key的话就是用到的索引名称，还有个预估扫描行数
  2. 失效的情况，因为采用索引是有个执行器进行决策的，主要看磁盘io成本和cpu成本，哪个低用哪个，如果数据量小的情况下io成本就比较低，就不会用索引，还有就是联合索引情况下不符合最左前缀匹配原则的话也会失效，还有就是在条件里面写运算导致每个id都要运算成本高就不采用索引，还有就是在里面写函数啊等等

RabbitMQ如何实现延迟队列的 ​

  1. 基于消息存活时间和死信交换机，就是把要延迟的消息不给他配置消费者，然后等到一定过期时间的话就自动绑定到死信交换机上丢到死信队列里面，然后再由消费者进行消费，过程一多时间久了就达到延迟效果
  2. 每个时间挡位各建一个队列过期时间相同就不会相互阻塞
  3. 延迟插件的本质是建一个用来存放延迟消息的队列空间-数据库，就是延迟的消息先在这个空间放置对应过期时间后在放回到队列中执行
