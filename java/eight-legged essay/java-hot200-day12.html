http1.0和2.0有什么区别 ​

  1. 就是围绕连接成本、队头阻塞、连接效率等
  2. 2.0解决了队头阻塞问题，一个连接阻塞其它连接都要等待、2.0就把每个请求打散成二进制传输流，到了服务端再组装
  3. 头部压缩，1.0的Head往往很大，每次传输都要带上传输效率就会被拉低，2.0后客户端和服务端都维护一个索引表，相同HeadId发送过来就可以复用
  4. 服务端感知，比如客户端需要一个index.html，服务端就猜到你应该还要sytle.css也一起发送过去
  5. 1.0的是纯文本协议，肉眼可见且传输较慢，2.0全面改为二进制传输，机器读取更快

消息队列如何保证消息不丢失 ​

  1. 就是生产、存储、消费三方面来
  2. 生产的话，发送消息到broke后，如果返回失败或超时了没响应就重试，同步发送的话就处理返回值，异步的话就在回调中处理失败情况
  3. 存储的话，要等broke持久化到硬盘上后再返回给生产者成功，集群情况下得同步到至少两个副本才算完成
  4. 消费的话要等业务逻辑真正执行完再进行消息确认，rabbitMQ最好手动确认，调用失败可决定重新入队或丢弃
  5. 可以专门给死信队列配一个消费者，然后消费的时候具体拉出来看原因

SpringAOP用的是哪种动态代理 ​

  1. 默认用jdk动态代理，如果目标类没有实现接口就用CGLIB动态代理
  2. 两者区别在于jdk动态代理是实现接口，然后生成代理对象类，调用代理方法时会反射调用真实对象上的方法
  3. CGLIB代理则是基于继承，运行时会生成目标对象的子类，子类重写父类具体方法，但是没法解决final类和final方法
  4. SpringBoot2.0后默认CGLIB

如何合理设置java线程池的线程数 ​

  1. 主要看是cpu密集型还是io密集型，cpu密集特点就是复杂计算与加解密，io密集就是文件读写数据库读写
  2. 如果是cpu密集则一般是cpu核心数 + 1，+1防止边界情况
  3. 如果是io密集则cpu核心数 * 2，让空闲出来的cpu可以去处理其他任务
  4. 以上只是参考，具体得根据业务情况不断去压测调整

redis中有哪些内存淘汰策略 ​

  1. 8种，分为不淘汰、淘汰有过期时间的，全部淘汰
  2. 不淘汰的话塞满就直接报错，然后锁住不让写入
  3. 有过期的话：1.随机淘汰设了过期时间的 2.淘汰存活时间最短的 3.淘汰最久没被访问的 4.淘汰访问频率最低的
  4. 全部的话：1.随机淘汰 2.淘汰最久没被访问的 3.淘汰访问频率最低的

mysql中我select * 一个一千万行数据的表，内存会飙升吗 ​

  1. 不会，因为mysql读取数据不会一下子加载到内存中，而是采用边读边返回的，也就是写满一定缓冲区就发送，然后清空缓冲区再写
  2. 客户端读的太慢导致本地网络栈被写满就写入暂停
  3. 排序和分组没法触发边读边返回，因为要先一次性排序和分组，就是全表扫描了

消息队列设计成推消息还是拉消息，优缺点？ ​

  1. 推消息由broke主动向消费者推送，实时性强，速度快，但是可能会和消费速度不匹配，导致堆积，消费者过载了
  2. 拉消息就是消费者主动向broke拉取，因为是自己拉，那么就会根据自身负载情况进行拉取，缺点就是可能会有延迟实时性不那么强，更适合批量处理
  3. 还有像roketmq和kafka的话就是拉消息，但是用了长轮询，就是拉取时如果有消息就立即返回，没就连接在那，等有消息就立即返回了

用过哪些并发工具类 ​

  1. conxxxHashMap:1.7使用分段锁，分成16段，每段单独加锁，1.8后使用CAS + sy锁细化到单个桶
  2. ReedtrainLock:相对于sy需要手动加锁和解锁，对于复杂业务场景更适用

什么是AOP ​

  1. AOP是面向切面编程，就是将多个模块的通用逻辑抽取出来，通过代理动态编织到目标方法上，一般是通过注解实现，像是权限校验、日志、事务等，通过代理对象调用真实对象的方法，还有前置通知和后置通知
  2. 切面：定义切入关注的地方如Spring AOP关注的是方法，切入点：定义哪些方法需要应用这个切面，比如com.xxx包下的所有方法，通知：具体切入了要做什么

什么是服务降级 ​

  1. 就是系统扛不住的时候砍掉一些非核心功能，只保留核心功能，比如商品详情页的推荐就返回一个热销榜单，不会说推荐服务挂了整个页面都显示不出来

redis的lua脚本有什么用 ​

  1. lua脚本就是一个命令，在redis服务端进行，保证了操作的原子性，比如加锁时要先判断锁是不是自己的，不是就删除，lua那边就把判断和删除打包在一起执行，避免过程中的插入其它操作，因为redis单线程，把lua脚本当作一个命令执行，期间不处理其它请求，看起来就有了原子性

http2.0和3.0的区别 ​

  1. 传输层改用udp协议，丢包时自己自动重试，而不会阻塞整个连接，只会阻塞丢包的那个
  2. 2.0需要tcp三次握手 + tls，3.0quic协议把传输层握手和加密握手合并了，如果是之前连接过的就可以直接发送数据
  3. 2.0ip变了就得重连，3.0基于一个id，ip怎么变都无需重连
