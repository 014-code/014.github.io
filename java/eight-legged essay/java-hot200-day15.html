线上发现redis机器爆了，如何优化 ​

  1. 首先用一些可视化工具来查看redis的cpu、内存、命令执行情况、带宽，确定具体原因
  2. 如果是内存原因就先去扩容内存，然后再看哪些数据占用了内存，是否需要存入redis，以及某些数据的过期时间是否过长
  3. 如果是cpu就可能是业务层面的了，一些读写操作频繁、耗时命令等，可以先以最小代价重新发版，退回上一版本，然后再去排查耗时的复杂业务逻辑是否可以优化等，大key拆成小key
  4. 带宽的话就加吧，可以本地缓存来分担压力
  5. 然后就是配置监控报警机制

说一下Spring Bean的生命周期 ​

  1. Spring通过反射调用构造方法创建Bean对象的空壳子
  2. 通过setter或字段反射将依赖的Bean注入
  3. 如果Bean实现了BeanNameAware等接口，Spring会将相应的容器信息注入
  4. BeanPostProcessor前置处理
  5. 初始化，按顺序执行@PostConstruct注解方法、@Bean指定的initMethod
  6. BeanPostProcessor后置处理，AOP代理通常在这一步生成
  7. 然后Bean就就绪可以用了
  8. 销毁容器关闭时执行@PreDestroy注解方法，@Bean指定的destroyMethod

Redis在生成RDB文件时如何处理请求 ​

  1. 首先客户端会发生一个生成快照的请求，主进程就fork出一个子进程-完成后就继续处理客户端的请求，这一步会有毫秒级的阻塞，然后主进程会将数据页表复制一份给到子进程，然后子进程遍历页表写入RDB文件，子进程写完后退出并通知主进程
  2. 快照期间数据可以改，主进程要改某个数据时，会将那个数据的数据页复制一份出来，主进程在副本中改数据，这样子进程看到的还是快照时刻的数据
  3. RDB到一半机器挂了怎么办：没什么关系，因为RDB时先写临时文件，写完再重命名为正式的，而重命名过程是原子的，要么全成功要么全失败

UDP和TCP的区别 ​

  1. TCP是面向连接的可靠流协议、UDP是不连接、不可靠的数据报协议，TCP适合准确性要求较高的场景-文件传输，保证不丢包，UDP适合实时性较强的场景、能够容忍少量丢包，如在线游戏、视频通话
  2. TCP拆包与粘包：粘包就是发送方发送的多个数据包被当成一个来接收了-因为TCP是字节流传输的没有明确的数据边界，没法区分消息边界，拆包就是接收方接收的一个数据包被拆成多个-因为可能是缓冲区大小超出了，接收方无法一次性接收到完整消息
  3. 定长消息发送、明确消息分隔符、消息头加一个能够接收数据量来告诉发送方

SpringMVC的具体工作原理 ​

  1. DispatcherServlet前端控制器拿取前端传入的url请求参数等，传入HandlerMapping去寻找带有@RequsterMapping注解的方法，匹配路径是否对应，如果对应则连同前端控制器和方法相关参数一同返回，接下来走Handler拦截器执行controller方法的前置处理，后面进入Handler执行器，执行controller方法，执行完再走拦截器执行后置方法，最后DispatcherServlet把结果转化成View视图对象返回给前端

分布式和微服务有什么区别 ​

  1. 分布式指的是一种运行方式，将一个任务分散到多个节点上执行，共同完成一个或多个共享任务
  2. 微服务指的是一种软件架构，将一个单体服务按照多个功能模块进行拆分，每个功能模块都能够松耦合得进行独立运行
  3. 微服务通常建立在分布式基础上，但分布式不等同于微服务，如果说把一个功能模块复制成10份来执行同一任务，是分布式但不是微服务

什么是java中的ABA问题 ​

  1. ABA是java中CAS操作的一个坑，就是CAS本质是比较并交换，他不关心你的值是否是原来的，只看值是否一致，就比如钱从100到50，又从50到100，CAS就成功，因为前后都是100，但是如果业务逻辑依赖这100是否是原来的，那就会出现问题
  2. 解决方法一般配合版本号去使用、还可以配合AtomicStampedReference这个原子类实现方式也是维护一个类似版本号的计数器，AtomicMarkableReference布尔值记录，就是只要被修改过就改成false然后后面就不能去修改了，适用于一些一次性场景

redis的哨兵是啥 ​

  1. 监控：每秒给redis所有节点发送ping，看下是否活着，故障转移：主节点挂了他会重新选取新主节点，然后将主节点地址通知给其它从节点
  2. 通常是用三个节点组成哨兵，相互通信一个挂了另一个也能顶上，然后也是读写分离，尽量单数，因为投票时不会平票

线程的声明周期 ​

  1. 首先new状态线程被初始化，还没调用start方法，然后调用start进入runnable方法，然后看锁抢到就继续runnable，抢不到则进入blocked阻塞，直到被唤醒，拿到锁就进入runnable，然后调用wait进入到wait状态，之后被唤醒回到runnable，再后面进入time——waiting，直到超时或被唤醒回到runnable，然后run执行

mysql建立索引需要考虑哪些内容 ​

  1. 不是越多越好，索引占据一定内存空间，越多需要维护越多B+树
  2. 区分度太低的不建议建立，没啥效果，字段太大的不建议，占用空间大
  3. 写多读少的表要慎重考虑，因为写多带来的索引负担反而会大于读的效益
  4. 排序分组的字段要建立索引，频繁查询的字段要建索引
