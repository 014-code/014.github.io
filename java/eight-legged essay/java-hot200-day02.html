mysql的索引越多越好吗 ​

  1. 不是，索引越多就会带来写操作负担，同时会带来内存、磁盘开销，总之就是会带来时间和空间上的开销
  2. 时间上：每次inster、update、deltete操作时都得去改被操作那个数据的主键索引，如果包含二级索引的字段还得去改二级索引，每个索引就是一个B+树，写的过多还会引发页分裂等
  3. 空间上：每个索引B+树的每个数据页就得存储16kb大小，索引一多就得存储大几百mb甚至几个gb，每次写的时候磁盘io次数就会很多
  4. 联合索引的字段顺序一般是：等值在前，范围在后，explain执行查询哪些索引被用了，然后有的无用索引可能是唯一键约束

B+树查询数据的全过程 ​

  1. 树的垂直查找，先从根节点查找，查询的键值和节点中索引键值比较，然后向下二分法查找顺着指针往下查找叶子节点
  2. 树的水平查找，页目录把叶子节点中的数据分成多个组，然后二分目录查找，然后在组内遍历链表即可
  3. Buffer Pool会缓存热点页数据
  4. B+树非叶子节点只存索引键值、指针一个页能容纳更多条目，且树更矮，利于减少磁盘io次数，且叶子节点间是以双向链表连接，io更具顺序性，不会像B树那样带来很多随机io

RabbitMQ的消息什么时候能绑定到死信交换机上 ​

  1. 消费者主动拒绝消息，requeue设置为false情况下，被消费者拒绝就会进入到死信队列，如果ture就会重新插入队列重试
  2. 过期时间到了还没被消费，ttl过期时间到了，这条消息还是没被消费就进入死信队列
  3. 队列满了，对头最老的消息就会进入到死信队列里

为什么选择B+树作为索引结构 ​

  1. B+树是多叉树非叶子节点不存数据，内存就能够存储更多索引项查询命中率高，就能在树高较矮的情况下存储更多数据
  2. 叶子节点间以双向链表连接，读写时io较为顺序，不必担心大量随机io带来的性能负担

RabbitMQ中无法路由的消息会去到哪 ​

  1. 默认情况下会丢弃，但是生产环境下肯定不行，1.备份交换机，对无法路由的消息转发到特殊的队列中专门用来存未被路由的消息，2.回退给生产者，让其监听回退的消息然后自主决定是要重发消息还是日志、告警等，3.可以使用死信队列作为兜底来临时存储
  2. 死信队列和备份交换机会冲突吗：不会，没被路由的都不会进到队列，死信是进了队列出问题，两者不一样

mysql的三层B+树能存多少数据 ​

  1. 2000万条左右，为啥：每个数据页默认大小16kb当然可以调整，假设每条数据为1kb，那么一个数据页就可以存储16条数据，那么一个非叶子节点中的指针6字节索引8字节，那么他就可以指向16 * 1024 / 6 + 8=1170个叶子节点，那么第三层就是1170 * 1170 * 16 = 2000w

sql语句在mysql中的执行过程 ​

  1. server层负责解析sql语句，连接管理等，引擎层进行数据的实际存取
  2. 过程：首先客户端先和数据库建立连接，而后验证账户与密码，之后进入分析器，会解析sql将其按照select where这这样的进行拆分，然后解析语法是否有误，然后生成一个抽象语法树，之后把语法树给到优化器，优化器会根据语法树决定要不要使用索引，要和那几张表join等，之后到执行器，会去调用存储引擎的接口一行行读数据，把结果返回给客户端，但是在mysql旧版本分析器前还会查缓存，但是表更新的话缓存就失效了，后来被砍掉了
  3. 走了索引查询还是很慢什么原因：1.回表次数太多，二级索引命中几十万行情况下，很多都要回查导致带来大量随机io2.索引字段选择太差，比如用性别这样的太泛了3.返回字段过多，没走索引覆盖4.索引树内存爆了，频繁走磁盘读取

mysql是如何实现事务的 ​

  1. 四大组件，undo log ，redo log、锁、mvcc
  2. redo log在事务提交前先写到redo log里面，保证事务提交过程中出现差错，也能通过redo log将数据回复
  3. undo log在修改数据前先把原值存入undo log，在事务回滚时要么全成功要么全失败
  4. 锁主要是行锁和间隙锁，行锁保证某一行数据正在被修改的情况下，不会被另一个修改操作所影响导致数据不一致间隙锁用来防止幻读
  5. MVCC保证隔离性的读写并发，读的时候不加锁，会通过版本链找到当时自己看到的数据版本，写的时候别人能读读的时候别人能写

事务提交两阶段 ​

  1. 其实是为了保证redo log和binglog的一致性
  2. 先在存储引擎层先写入redo log，然后打标记-此时还未真正提交事务，然后再去server层写入binglog，并通知引擎层将状态改为commit，提交事务
  3. 为什么需要二阶段，就是怕其中一个宕机另一个就能够恢复他的数据，还要检查两边数据是否一致，不一致回滚

mysql长事务带来的问题 ​

  1. 锁竞争严重，长事务一直持有行锁和间隙锁，导致其它事务等待且上游被阻塞，业务线程会被阻塞，同时产生循环等待产生死锁的概率增大
  2. 事务回滚的代价，长事务要回滚的话，期间所做的任何操作也得一并回滚，之前做的操作可能就白白浪费了
  3. undo log膨胀，undo log也得记录事务一开始的状态，且随着时间增长版本链只会越来越长，导致内存占用不断变大
  4. 主从延迟，执行长事务积累到binlog中需要10分钟的话，备份到从库也得需要10分钟，时间成本直线增加
  5. 需要活跃事务读这些版本链上的某个节点时才会去开线程清理旧版本节点

什么是MVCC ​

  1. 俗称多版本并发控制,就是让读写操作互不阻塞
  2. 就是在innoDB引擎层中每条数据都会记录，最后一次修改该数据的事务id和当前指向undo log某一版本的指针，每次修改数据操作的时候先写入undo log记录，然后逐步形成版本链，读的时候就根据当前时间匹配到最近的一个可见版本节点
  3. 这就做到了读各自的写各自的这种无锁操作，并发性能比较强
  4. 可重复读不能避免幻读，幻读本质上是两次读取的行数不同，在快照读完后有其它事务进来修改了数据，再当前读读到的就是修改后的了，行数就可能会不同
