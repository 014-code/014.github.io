什么是tcp连接 ​

  1. 是一种传输层面的协议，保证数据从一台机器到另一台机器的可靠传输
  2. 本质是状态同步，连接双方都维护一些状态，包括要发送的ip、端口，接收方ip、端口
  3. 还有存储对方目前收到的字节数，窗口大小告诉对方我还能存多少

消息队列如何处理重复消息-保证消息的幂等性 ​

  1. 首先想到的是建立一个队列任务表，然后给每个任务都分配一个uuid--生成者端，消费任务前先查数据库有就直接返回，没就插入数据--消费者端
  2. Borke端的话就是会自动去重，也对应前面生产者分配的每个任务id
  3. 然后同样的也适用于redis-消费者端
  4. 然后就是乐观锁了，消息加版本号，还有设计流转状态，还有就是可以使用业务单号作为唯一索引，比如订单号--下游业务的存储端

Java中死锁产生的条件以及如何避免死锁 ​

  1. 循环等待，两个线程在互相等待，不可剥夺，一个线程占有资源时不可被其它线程剥夺，资源同一时刻只能被一个线程占有，一个线程占有一把锁的同时还想要另外一把锁
  2. 固定加锁顺序，不要去越级抢锁，使用ReentrainLock的tyrLock获取锁，超时则自动放弃锁
  3. 数据库层面也会产生死锁，两个事务互相锁住对方需要的行，但是mysql会自动检测死锁以最小代价回滚
  4. 活锁就是不阻塞但是一直做无用功，线程饥饿就是一个线程长期得不到资源，可用公平锁，ReentrainLock构造时传一个true就是了

消息队列如何保证消息的有序性 ​

  1. 先来说无序性：就是消息被分发到不同队列，被不同消费者消费
  2. 那么就从这上面来看，我只让他分发到一个队列上对应一个消费者就可以解决，但是这样效率太低-单生产、单队列、单消费
  3. 那么再针对消息本身，可以指定相似/相同属性的消息发送到相同队列,然后每个队列一个消费者，构造key之类的

什么是springioc ​

  1. 就是一个自动控制对象创建及其依赖关系的容器，以前是收到在A里面new一个B，ioc是，A告诉容器我需要B，那么把B注入A，ioc容器会识别到这个注入，然后在A中创建对象B，至于用B的哪个实现类等都交由ioc自动处理，也就是控制翻转，翻转的是对象创建的控制权，控制权交给ioc了
  2. 构造器注入、setter注入，字段注入，spring新版本推荐构造器注入，因为构造器字段可以声明为final不可变,依赖不会为空，参数缺失就直接报错，不会后面再空指针
  3. 两个注解一个是按类型匹配一个是按名称匹配，用哪个都差不多，@Autopwrite属于Spring原生的

什么是线程池 ​

  1. 就是统一管理线程声明周期的一个工厂吧，用线程池避免线程的频繁创建和销毁的开销，因为线程池默认不会直接创建线程而是等任务来了才创建，懒创建，但是也可以预热
  2. 核心线程数，最大线程数、工作队列、空闲存活时间、拒绝策略
  3. 核心线程数满了先丢到工作队列中排队，后面才开始创建非核心线程，然后都满了达到最大数就触发拒绝策略，闲下来的线程超过空闲存活时间且非核心超过核心的话就回收掉多余的，核心线程一般不会回收但是也可以配置
  4. 拒绝策略的话：直接抛异常-适合核心业务自己手动处理，踢掉最久的那个线程，静默丢弃，调用者线程拉去干活自然提交不了新任务，还可以自定义策略

redis过期数据的删除策略是什么 ​

  1. 惰性删除 + 定时删除
  2. 读某个可以前先看是否过期，过期就删除返回空，定时就是定时任务去检查过期key删除，不是一次性删除，部分删根据比例和时间
  3. aof日志重写时也会检查过期key，过期的就不写入了

消息队列如何处理消息堆积 ​

  1. 无非就是生产者生产消息的速度远大于消费者的消费速度
  2. 那么就增加消费线程数或消费者示例，也有可能是消费逻辑代码执行较慢可以适当优化，可临时拓展多个消费者队列，积压消息消费完就关闭
  3. 然后就是针对消息本身，对不重要的消息可以适当延迟处理或部分舍弃，保留核心消息-消费者降级方面
  4. 还有就是对生产端限流/降级，控制生产速度

什么是服务熔断 ​

  1. 就相当于保险丝，下游服务扛不住时调用方主动切断服务，如在10s内失败超过50%就切断，等一段时间后放几个请求试探，可以就恢复，降级的话就是止损后返回的内容，一个兜底的响应
