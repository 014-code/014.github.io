Q: 在学期→课程→技能树→技能点四级数据上，LinkedHashSet 维护节点顺序，LinkedHashMap 建立链路关系，实现专业系数 × 点亮值 × 权重的加权计算，定时任务计算各年级技能树均值

A: 1. 为什么用 LinkedHashMap 而不是 HashMap? A: 因为桑基图需要按层级顺序展示，从左到右是 学期→课程→技能树。HashMap 的底层是哈希表，遍历顺序是哈希码随机顺序，无法保证数据顺序 A: 3. LinkedHashMap 底层原理？ A: LinkedHashMap 继承 HashMap，内部额外维护了一个双向链表（head/tail 头尾指针，每个 Entry 有 before/after）。每次 put 时会调用 linkNodeLast() 把新节点加到链表尾部，遍历时按双向链表的 before 指针顺序返回元素，所以能保证插入顺序 A: 5. 节点去重怎么实现的？ A: 一个课程可能关联多个技能树，同一节点在不同链路中出现，所以需要用 LinkedHashSet 去重 A: 7. 桑基图数据需要什么格式？ A: 从左到右：学期 → 课程 → 技能树，表示学生的专业课成绩与技能树点亮的流向关系nodes（三个层级的所有元素、三级技能点包含点亮情况）+ links（中的哪个元素指向哪个元素） A: 9. 加权计算 A: 已点亮分 + (总分 - 已点亮分) × (课程均分 / 100) × 权重 = 最终得分

Q: 草稿隔离 + 技能树 version 乐观并发控制树级结构，ReentrantLock 实现多人编辑技能树防冲突

A: 1. 你说用 ReentrantLock 防冲突，那 ReentrantLock 和 synchronized 有什么区别？为什么不用 synchronized？ A: ReentrantLock 可中断、可超时、公平锁，synchronized 简单 A: 3. 版本号是每次修改 +1，那如果两个人同时拿到版本号 1，都去修改，谁先成功？失败的那个怎么办？ A: 用户A 先发布 → 检查版本号 → 成功 → 正式表更新，版本号+1，用户B 后发布 → 检查版本号 → 发现已变成 2（比自己手中的 1 大）→ 提示'已被修改，请重新加载' A: 5. 草稿隔离如果两个人同时编辑同一棵树的同一个节点，最后发布以谁为准？ A: 谁先提交以谁为准 A: 7. 如果服务器突然重启，ReentrantLock 的内存锁是不是就没了？那怎么办？ A: 所以数据库记录版本号兜底 A: 9. 草稿数据会不会越来越多？有没有清理机制？ A: 成功发布后草稿就被删除了 A: 11. 草稿隔离这种方式真的有必要吗？直接加锁不就行了，还要搞两套表，增加复杂度，意义在哪？ A: 因为会有多人编辑的场景，ReentrantLock控制同一棵技能树同时只有一个线程在发布

Q: 调用公司内部 Dify 工作流与 RabbitMQ 异步解耦配合消息重试实现ai面试/对话、个性化建议

A: 1. RabbitMQ 怎么保证消息不丢失？生产端、消费端、broker 分别要做什么？ A: 生产端 confirm + 消费端 manual ack + 持久化 A: 3. 如果消费者处理消息时抛异常了，消息会怎么办？你说配合消息重试实现，那重试机制怎么设计的？是消费失败后立即重试吗？ A: 在我们的项目中，消息消费失败后会检查重试次数。如果未超过 3 次，消息会重新入队等待下次消费；如果超过 3 次，设置任务状态为失败，然后 basicNack(requeue=false) 拒绝消息。同时我们用数据库记录了任务的重试次数，消息体里也带了 num 字段，可以精确控制重试次数 A: 5. 顺序性？ A: 单队列 + 单消费者，或消息序号 A: 7. 如果消费者一直消费失败，消息会不会无限重试？怎么防止？重试次数用完了还是失败，消息怎么办？ A: 在我们当前的项目中，重试 3 次后消息就被拒绝了，没有走死信队列。消息会被丢弃，同时数据库任务状态会标记为失败（状态 2），后续由人工处理 A: 9. 如果 Dify 工作流处理不过来，消息积压了怎么办？ A: 增加消费者实例、增加消费者数量、优化消费代码逻辑减少io操作或使用批量处理、设置消息优先级对非关键消息丢弃或延迟处理、生产端限流降低生产速度 A: 11. 批量处理的问题：顺序性（按消息的业务主键（比如 taskId）分组，保证同组顺序）、部分失败问题（事务 + 失败跳过/重试）、幂等性问题（业务唯一ID + 数据库唯一索引）、控制批量大小

Q: 在用户点亮技能、每日登录等提升等级行为采用事件驱动处理，Redis SETNX 幂等增加经验值

A: 1. 事件驱动具体怎么实现的？Spring 的 EventListener？和观察者模式什么关系？ A: Spring @EventListener，发布订阅模式 A: 3. Redis 防重复，用的什么命令？SETNX？那和 SET 有什么区别？ A: GET + SET 是两步操作，非原子。高并发下可能出现两个请求同时查到 key 不存在，然后都去发放经验值，导致重复发放。SETNX 一步原子操作，key 不存在才设置成功，能保证并发安全 A: 5. 为啥要用SETNX？ A: 事件监听是异步执行的。而且前端如果没做防抖，用户快速连续请求两次，就会触发两次事件下发经验值。虽然现在还没出现重复发放的问题，但为了保证高并发下的准确性，使用 SETNX 原子操作，这是更稳妥的方案 A: 7. SETNX如何保证原子性？ A: "SET if Not eXists"，key 不存在才设置，Redis 只有一个线程处理所有命令，不会并发一个命令执行过程中不会被其他命令打断

Q: 基于 WXJava 集成微信支付全流程(支付/退款/商户转账)、事件驱动 + @Async 异步推送微信订阅消息、代理商与平台分润

A: 1. 微信支付流程说一下？ A: 用户下单，后端调用统一下单接口 → 获取 prepay_id，返回 prepay_id 给前端，前端调起微信支付 → 用户输入密码，微信回调后端 → 验签 → 更新订单状态，WXJava 内部处理：获取验签需要的参数（签名、时间戳、随机串等）用商户私钥生成签名，与微信返回的签名比对 A: 3. 支付状态同步问题？ A: 我们用回调通知为主、主动查询为辅。支付成功时微信会回调我们的接口，如果回调失败或者超时，我们会定时任务主动查询微信订单状态，保证不丢单 A: 5. @Async 异步消息会丢失吗？ A: 我们用的是 Spring 的 @Async，默认用线程池。如果方法执行到一半应用挂了，消息会丢失。但因为是通知类任务，丢失影响不大。如果要保证不丢，可以用消息队列，@Async在一个类内部调用会this就失效，还有public才行且必须是Spring进行管理的 A: 7. 分润逻辑怎么设计的？ A: 两种情况：用户没绑定代理商 → 平台全收：systemCommission = 订单金额 × 平台抽成比例；用户绑定了代理商 → 代理商抽走一部分：actingCommission = 订单金额 × 代理商抽成比例；systemCommission = 订单金额 × 平台抽成比例 × 代理商抽成比例 A: 9. 分润对账怎么做的？ A: 我们每天跑定时任务，把订单表里的分润数据和微信转账记录对账。如果对不上就告警，人工处理 A: 11. 支付成功后又退款，分润怎么处理？ A: 退款时会把分润金额扣回来。退款成功后，调用分润冲正接口，把之前结算给代理商的钱扣掉 A: 13. 如果让你设计一个支付系统，你会怎么做？ A: 保障 idempotency：用商户订单号唯一索引，保障一致性：先更新本地状态，再请求微信，保障不丢单：回调 + 主动查询双重保障，保障资金安全：所有请求都要验签，异步化：支付回调后异步更新状态、快速返回

Q: 基于 MySQL 版本号乐观锁 + TransactionTemplate 编程式事务确保订单/支付状态流转的原子性

A: 1. 为什么用编程式事务而不是声明式 @Transactional？ A: 订单状态流转涉及多个表的更新（订单表、账户表、日志表等），逻辑复杂。用 TransactionTemplate 可以更精细地控制事务,手动回滚等 A: 3. 乐观锁原理是什么？ A: 更新订单状态时查出版本号并where带上返回0行说明被改了自然update也不会成功 A: 5. 失效情况？ A: 本质上是aop，final、public、同一个类中调用会绕过aop代理、抛出的不是 RuntimeException 默认不会回滚 A: 7. 为什么要同时用编程式事务 + 乐观锁？ A: 用途不同：TransactionTemplate 保证多个表操作要么全成功要么全失败（事务原子性）；乐观锁 保证单条记录不被并发重复修改（数据一致性）；两者配合，一个管多表事务，一个管单表并发

Q: 基于 LangChain4j 集成 DeepSeek，支持 Vue/React 项目生成，门面统一 AI 调用，策略模式/模板方法完成代码解析、持久化

A: 1. 为什么要用策略模式切换 Vue/React 项目生成？模板方法用在哪？ A: 不同项目类型需要不同的 AI 配置：Vue/React 需要文件读写工具（Tools），用流式输出；HTML/多文件 直接生成完整代码，不需要工具；用策略模式可以根据 codeGenType 动态选择 AI 服务实现。模板方法由父类定义代码生成流程、子类继承实现具体生成逻辑 A: 3. 门面模式统一 AI 调用怎么做的？ A: 门面（Facade）统一了所有 AI 调用的入口，屏蔽了内部策略的差异。上层业务不用关心用哪个模型、什么配置，根据代码生成类型从代码生成工厂类中获取对应代码生成服务对象service A: 5. AI 生成代码出错了怎么办？ A: 我们做了：工具调用次数限制（10 次），输入安全校验（Guardrails），幻觉工具处理（不存在的方法返回错误），LangGraph 工作流里还有质量检查节点，不合格会重试 A: 7. 多个用户同时生成代码，AI 实例会很多吗？ A: 用了 Caffeine 缓存 AI 服务实例，同一用户同类项目复用同一个实例，避免创建太多对象。30 分钟无访问自动过期

Q: Redis 持久化对话历史，区分memoryId多用户会话隔离，RRateLimiter + AOP 实现用户/IP/接口级限流

A: 1. 对话历史怎么实现多用户隔离的？ A: 在创建应用服务的工厂类里面的创建应用生成服务方法里面会将appid传入，然后创建一个消息上下文对象memory将appid作为他的memoryid，然后使用配置好的redis对话存储方法进行存储这个appid下的对话消息 A: 3. 对话历史有 TTL 吗？怎么设置的？ A: 配置了 24 小时 TTL。超过时间不活跃的会话自动过期，节省 Redis 内存。 A: 5. 对话历史太长怎么办？ A: 配置maxMessages只保留最近 30 条消息。超过的自动丢弃 A: 7. Redisson实现限流的过程? A: 先定义了一个@RateLimit注解，需要增加限流接口加上，里面有参数如什么类型限流令牌数大小等，然后redis生成唯一一个限流key，用户是user:id，ip则是ip:id，然后每个key都有自己的令牌桶，系统以一个固定速率往里面放令牌，请求打过来后取令牌，令牌没了就拒绝抛出自定义的业务异常 A: 9. 滑动窗口和令牌桶区别？ A: 滑动窗口统计固定时间窗口内的请求数缺点是要维护大量的时间戳，令牌桶更适合允许突发的场景，比如用户狂点按钮后恢复正常频率。滑动窗口适合严格限制，比如接口绝对不能超过 QPS

Q: 虚拟线程异步完成 npm 构建，Process API 执行项目打包，编写ESLint检查工具保证代码生成可运行性

A: 1. 为什么用虚拟线程做异步构建？ A: 轻量级：由 JVM 管理，不占用 OS 线程；高并发：可以轻松创建数十万个虚拟线程；低成本：创建/销毁几乎无开销；更适合io密集型任务，虚拟线程在阻塞时会自动"卸载"（unmount），让出底层平台线程。这样，一个平台线程可以运行很多虚拟线程，大大提高并发能力 A: 3. 构建超时怎么办？ A: waitFor等待60秒，时间到了自动杀掉进程 A: 5. ESLint 怎么集成的？ A: 作为 LangChain4j 的 Tool 暴露给 AI：AI 生成完代码后调用 validateCode 工具自动安装 ESLint 依赖执行 npm run lint 检查返回检查结果 A: 7. ESLint 怎么自动安装的？ A: AI 生成的是 Vue 还是 React 项目不确定，所以：先判断项目类型（检查 package.json 依赖），动态添加对应的 ESLint 插件，自动执行 npm install A: 9. 虚拟线程的原理了解吗？ A: 虚拟线程是直接交由jvm管理，不运行在平台os线程上，jvm内部维护一个类似虚拟线程池的，一个平台线程可挂载多个虚拟线程，当一个虚拟线程阻塞时，JVM可以立即切换到另一个虚拟线程，而无需等待操作系统的调度，虚拟线程不需要一个完整的线程栈，所以占用创建销毁开销极低

Q: 使用 Websocket 实现多人协同生成应用/多人可视化编辑，Disruptor 无锁队列异步处理ws消息

A: 1. Disruptor 和 RabbitMQ 对比？ A: Disruptor 是进程内的高性能队列，适合单机内异步化、低延迟场景；RabbitMQ 是跨服务消息队列，适合微服务通信、需要持久化、削峰填谷 A: 3. 消息顺序性怎么保证？ A: Disruptor 本身是顺序的，加之单生产者单消费者，整个操作就是有序的了 A: 5. Disruptor 的 RingBuffer 大小怎么选？ A: 2 的幂次：方便位运算取模，不能太小：会频繁覆盖，不能太大：内存浪费，我选用的是256K A: 7. 为啥要用Disruptor？ A: 传统线程池锁竞争激烈，Disruptor是一个环形队列，采用CAS加内存屏障实现无锁操作，并维护一个固定大小的内存缓冲区利于内存复用

Q: 利用 postMessage API 实现主应用与 iframe 中预览站点的跨窗口通信，动态注入监听脚本实现可视化编辑网站

A: 1. postMessage 是什么？解决什么问题？ A: postMessage 是 HTML5 提供的 API，用于跨窗口/跨域通信，不同域名的页面无法直接访问对方 DOM，第二个参数指定接收源限制，*表示所有也可以指定某个域名 A: 3. 怎么往 iframe 里注入脚本？ A: 首先通过vite的代理配置保证主页面和iframe处于同源，进入编辑时父页面会创建一个js脚本文件注入到iframe中的dom，被注入的脚本中包含点击事件的监听器，去监听可视化编辑，点击之后会阻止默认事件与冒泡，将被点击元素打包发送给父页面，父页面同样监听执行逻辑 A: 5. 跨域安全问题怎么考虑？ A: 发送时指定目标 origin、接收时验证 event.origin A: 7. 怎么实现多人协同编辑？ A: 一个用户的iframe点击后发送给父页面，父页面收到后广播消息给另一个用户，另一个用户的父页面收到后发送消息给其子页面，然后子页面高亮显示选中元素

Q: 在投票模块的业主专属投票接口中，基于 Stream API 对重复数据过滤 groupingBy 分组计算各类型票数与投票面积，保障投票结果准确性，@RepeatSubmit 进行防抖

A: 1. 你说用 Stream API + groupingBy 分组计算，能讲讲具体怎么实现的吗？ A: groupingBy 本质上是 Map 分组，key 是 optionId，value 是该选项下的用户列表，然后用二次分组计算面积，这里用 result + "-" + isTakePartIn 作为分组条件，计算每种状态的面积总和 A: 3. 为什么要用 result + isTakePartIn 两个字段组合作为 key？ A: 因为投票有多种状态：1-0 同意且已投票；2-0 反对且已投票；9-0 弃权且已投票；0-1 参与投票（未投票但报名了）。两个字段组合才能唯一确定一种投票状态 A: 5. @RepeatSubmit 是什么？怎么实现的？ A: 生成唯一 key（接口路径 + 参数 hash + 用户 ID），存到 Redis，key 有过期时间，第二次请求时检查 Redis 中是否存在，存在则拒绝，不存在则放行并写入 Redis A: 7. 如果不用 @RepeatSubmit，有什么替代方案？ A: 前端防抖：loading状态加点击生成定时器，页面刷新会失效；数据库唯一索引：报错需回滚；请求ID去重：相同ID拒绝、前端需生成唯一ID A: 9. 你们投票结果是怎么保证准确性的？ A: 按房屋面积算权重，不是按人数

Q: 通过断点排查 + JUnit单元测试修复了小区楼栋查询遗留bug，Aspose 进行投票结果的pdf导出

A: 1. JUnit 单元测试你们主要测哪些场景？ A: 边界条件：空值、超长、分页；Mapper 层 SQL：复杂查询是否返回正确数据 A: 3. 如果单元测试跑不过，你会怎么排查？ A: 看断言期望值是否正确，看测试数据是否和实际场景一致，加日志输出中间结果 A: 5. 你们投票结果是怎么导出 PDF 的？ A: 先用 Apache POI 生成 Excel，再用.Aspose 把 Excel 转成 PDF。直接用 iText 或 PDFBox 生成 PDF 很难控制格式，尤其是复杂表格。Excel 是现成的模板，Aspose 转换后格式基本不变，更省事 A: 7. 导出大文件时要注意什么？ A: 内存：大 Excel 会 OOM，建议用 SXSSF（流式写入）；样式：缓存样式对象，不要每次创建；模板：只读一次模板，复用到所有 sheet；并行：可以用 parallelStream 提升性能

A: 1. 你们为什么要在 (option_id + del_flag + result + house_id) 上建联合索引？ A: 因为拆分后的各类查询条件都覆盖这几个字段，且联合索引可以覆盖查询，避免回表。同时 del_flag = 0 是固定过滤条件，放在中间能利用索引 A: 3. 联合索引什么时候会失效？ A: 跳过最左列、中间列缺失、使用函数、类型转换 A: 5. 索引顺序怎么设计的？为什么 del_flag 放第二个？ A: 最左前缀原则：索引 (A, B, C) 可以支持 A、A+B、A+B+C，但不能支持 B+C。所以把高频固定条件放前面。 A: 7. 拆分后怎么保证数据一致性？ A: 一次性查出所有需要的数据到内存，用 Java Stream 在内存中分组、聚合保证同一批数据计算结果一致 A: 9. 20秒优化到5秒，你是怎么定位到性能瓶颈的？ A: 慢查询日志：开启 MySQL 慢查询，看具体哪条 SQL 慢；EXPLAIN 分析：看执行计划，是否全表扫描、索引失效；断点 + 日志：打点记录每个方法耗时 A: 11. EXPLAIN 关注哪些字段？ A: type：最好 const/range，避免 ALL（全表扫描）；key：实际用的索引；rows：扫描行数，越少越好；Extra：Using filesort/Using temporary 不好 A: 13. 如果让你重新设计，你会怎么优化投票预览？ A: 分批处理-1000条一批；保证分批数据一致性：同一事务（批量操作在同一个 @Transactional 里）、游标分批（每次查询范围确定，结果可预期，不依赖数据总量，缺点：无法跳页，先查第一页后有新数据无法感知第二页时会被重复查）；每个投票记录的版本号/时间戳，批处理前先快照读版本号，查的时候校验有不一样则重新查，但是我们的系统中变更频繁这样反而效果不好
